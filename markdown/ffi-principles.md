1. hard 80/20 rule, 80% of stuff should be dumb cases and handled automatically 20% should be rust/zig style annoying explicitness
2. 80/20 applies to implementation too, if something is complex advanced a "really nice feature" so to say, its probably in that 20% category and should be handled by the programmer and not via a compiler state machine or whatever
3. keep it safe and honest, almost to same degree is as elm and rust. this also means you cant just be passing opaque stuff around because then why even have types?
4. absolutely no reimplementing typescript, no fixed point looping etc. thats what the typescript reflection part is for. same with if only way to figure something out is some super advanced special system, it should just fail and require the user to be more explicit
5. the ffi system exists for making the language immidiately useful, it should not be exciting or theoretically satisfying, nor solve every problem because after all im working on this language because i wish typescript had less features, not more
6. the ffi boundary should translate foreign programming models into workman's programming model, not extend workman until it can directly express every host-language concept. failure to reflect an api is not by itself evidence that workman needs features such as foreign subtyping. when an api fundamentally does not fit, a typed shim should contain that mismatch and expose a smaller fp-shaped interface. typescript checks the foreign side of the boundary and workman checks the interface that crosses it.
7. js.object and js.value describe genuinely dynamic foreign values; they are not recovery types for reflection failures. if typescript provides a static type and workman cannot map it honestly, keep the ffi obligation unresolved and report that unsupported type. silently replacing known structure with an opaque type destroys evidence that hm cannot recover and usually only moves the failure downstream. dynamic values should enter js.object/js.value because the declaration is actually dynamic or because the programmer explicitly chose that boundary, and leave it through an explicit assertion or other checked operation.
